# Dev-note: fixed-point `Biquad` — CLOSED in P10 (was a deliberate `@compileError`)

**Status (updated P10, 2026-06-04): RESOLVED.** `src/filters.zig` `Biquad(num)`
now supports **integer/fixed-point lanes** too. The former `@compileError` on the
integer path is gone; `Biquad` dispatches to `BiquadFloat` (DF2T, f32/f64,
unchanged) or `BiquadFixed` (DF1, any integer lane). A q15 (`i16`) section is
bit-exact against an independent NumPy oracle (`tests/gold_fixedpoint_test.zig` +
`tests/vectors/biquad_q15.json`, generated by `scripts/generate.py`'s
`_fix_biquad`).

> The rest of this note is retained as the rationale for *why* it was hard and
> *how* it was closed.

**Former status:** `Biquad(num)` supported **float lanes only**; an integer
precision was a compile error by design.

## Why it is NOT a straight q-format port

A biquad's coefficients break the simple "multiply, `>>frac`, saturate" mould the
`Gain`/`ConstantPowerPan` integer paths use:

1. **Coefficients exceed unity.** A resonant section has `a1 ≈ -1.9`, `a2 ≈ 0.95`.
   A plain q15 number only spans `[-1, 1)`, so `a1 = -1.9` **cannot be stored at
   all**. A real implementation needs a wider coefficient Q-format — e.g. CMSIS-DSP's
   `arm_biquad_cascade_df1_q15` uses **Q1.14 coefficients + a `postShift`** so
   `|coeff|` up to ~2 (or ~4) is representable.
2. **Accumulator headroom.** The 5-term MAC `b0·x + b1·x₁ + b2·x₂ − a1·y₁ − a2·y₂`
   must accumulate in a strictly wider type (i64 for a q15 section) with guard bits,
   then a single rounding shift + saturate on store. A per-multiply `>>frac`
   (as `Gain` does) loses the inter-term headroom.
3. **Limit cycles.** Quantization *inside* the feedback loop can drive small-signal
   self-oscillation (a non-decaying "limit cycle") that the float path never
   exhibits. Mitigation (error feedback / dithered quantization in the loop) is part
   of a correct fixed-point IIR, not an afterthought.

A naive q15 biquad would therefore compute **silently wrong audio** for ordinary
filters. Per the project's fail-loud rule, the kernel refuses to compile rather than
ship that.

## How it was closed (P10, the embedded-precision phase)

Implemented as `BiquadFixed` in `src/filters.zig`:

- **Coefficient format: `Q(2.cf)`, `cf = bits - 3`** (q15 → Q2.13, range [-4, 4)),
  carried in `Coeffs(T)` (now lane-aware: integer identity `b0 = 1 << cf`). This is
  WIDER than the lane's own q-format, so a resonant section's |a1| up to ~2 is
  representable — the exact thing q15 could not hold. (Simpler than CMSIS's
  Q1.14 + `postShift`: a fixed Q2.13 needs no per-section shift and stays
  bit-exact-reproducible.)
- **Accumulator: a wide `std.meta.Int(.signed, 2·bits + 4)`** (i36 for q15, which
  the backend lowers to i64). The 5-term MAC sums there with full inter-term
  headroom; a single round-to-nearest right shift by `cf` + saturate happens only
  on store. (A per-multiply `>>frac`, as `Gain` does, would lose the headroom.)
- **Direct form I (4 state words: x1,x2,y1,y2), state at the lane format.** The
  feedback path sees the clean quantized q15 signal, not a lossy intermediate —
  the DF2T form's accumulator-scale state would worsen limit cycles in fixed point.
- **Bit-exact gold:** `_fix_biquad` in `scripts/generate.py` mirrors the integer
  arithmetic exactly (Python ints = unbounded acc; arithmetic `>>` matches Zig
  signed `>>`); the manifest carries pre-quantized integer coeffs `b0_q…a2_q`, so
  no transcendental enters the comparison. `tests/gold_fixedpoint_test.zig` asserts
  bit-for-bit equality.
- **Stability/limit-cycle check:** `src/filters.zig` has an impulse-decay test
  (the tail magnitude must fall below the early response — no runaway / sustained
  self-oscillation); deeper Yoneda coverage is dispatched at the P10 gate.

**Note (a deviation from the plan's wording, Rule 7/12):** the plan §10 work-item
said "q15 saturating MAC in `i32` `Acc`," but a sum of five (lane × supra-unity
coeff) products overflows i32; this note's earlier reasoning (i64) is correct and
is what shipped. `num.Acc` (i32 for q15) remains the right MAC width for `Gain`'s
single product — the biquad's specific 5-term MAC locally widens beyond it.

## What already works in fixed-point (unchanged)

`Gain(q15)` and `ConstantPowerPan(q15)` **do** run — they are single multiplies with
`|coeff| ≤ 1`, representable in q15, validated by **live bit-exact gold vectors**
(`tests/gold_fixedpoint_test.zig`, manifests `tests/vectors/gain_q15.json` and
`pan_q15.json`). The panner takes pre-quantized integer gains via `gains_q` so its
fixed-point path carries no runtime trig and is bit-exact-testable. So fixed-point
is not absent — only the one block where it is genuinely hard is gated off.
