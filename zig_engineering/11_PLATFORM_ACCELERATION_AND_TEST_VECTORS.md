# 11: Platform Acceleration, Optional Libraries, and the Test Vector System

## 1. platform.zig — Best-Effort Dynamic Acceleration

`src/core/platform.zig` (64 lines) holds two mutable globals — `libs` (the optional
acceleration libraries) and `debug` (the debug-print flag) — plus the
`initialize`/`waitForInterrupt` entry points.

```zig
pub var libs: struct {
    liquid: ?std.DynLib,
    volk: ?std.DynLib,
    fftw3f: ?std.DynLib,
} = .{
    .liquid = null,
    .volk = null,
    .fftw3f = null,
};

pub var debug: struct {
    enabled: bool,
} = .{
    .enabled = false,
};

pub fn initialize(allocator: std.mem.Allocator) !void {
    debug.enabled = try lookupEnvFlag(allocator, "ZIGRADIO_DEBUG");

    if (libs.liquid == null and !try lookupEnvFlag(allocator, "ZIGRADIO_DISABLE_LIQUID")) {
        libs.liquid = std.DynLib.open("libliquid.so") catch null;
    }
    if (libs.volk == null and !try lookupEnvFlag(allocator, "ZIGRADIO_DISABLE_VOLK")) {
        libs.volk = std.DynLib.open("libvolk.so") catch null;
    }
    if (libs.fftw3f == null and !try lookupEnvFlag(allocator, "ZIGRADIO_DISABLE_FFTW3F")) {
        libs.fftw3f = std.DynLib.open("libfftw3f.so") catch null;
    }
}
```

The lookup mechanism is `std.DynLib.open` on the bare Linux SONAMEs
`libliquid.so`, `libvolk.so`, and `libfftw3f.so`, each guarded by a
`ZIGRADIO_DISABLE_<NAME>` environment flag. `open` failures are swallowed with
`catch null`, so a missing library simply leaves the slot `null` — initialize never
errors out over an absent accelerator. The env-flag parser (`lookupEnvFlag` →
`isTruthy`) treats `"1"`, `"true"`, `"TRUE"`, `"yes"`, and `"YES"` as true. The
same file also exposes `waitForInterrupt()`, which blocks on `SIGINT` via
`std.posix.sigprocmask` + `std.c.sigwait`.

Blocks that can benefit query the populated globals directly. The real idiom (see
`src/blocks/signal/multiply.zig`, `complexmagnitude.zig`, `frequencytranslator.zig`,
`iirfilter.zig`, etc.) is to pick an implementation at `initialize` time:

```zig
if (platform.libs.volk != null) {
    self.impl = .{ .volk = .{} };
} else if (platform.libs.liquid != null) {
    self.impl = .{ .liquid = .{} };
} // else fall back to the pure-Zig impl
```

and then resolve the actual symbols on first use, e.g.

```zig
volk_32fc_x2_multiply_32fc = platform.libs.volk.?.lookup(
    @TypeOf(volk_32fc_x2_multiply_32fc), "volk_32fc_x2_multiply_32fc"
) orelse return error.LookupFail;
```

VOLK is preferred over liquid-dsp when both are present. FFTW3f (`libfftw3f.so`) is
loaded the same way for future FFT-based blocks. Note that the SDR I/O blocks
(`sources/rtlsdr.zig`, `sinks/pulseaudio.zig`, `sources/airspyhf.zig`) open their
*own* dynamic libraries (`librtlsdr.so`, `libpulse-simple.so`, `libairspyhf.so`)
rather than using `platform.libs`.

This keeps the signal-processing core pure Zig and dependency-free by default, while
giving users who have the accelerator libs installed a free speedup.

## 2. The vectors/ Directory — Generated Oracles

`src/vectors/` mirrors the source tree: `src/vectors/utils/{filter,window,sample_format}.zig`
plus `src/vectors/blocks/{signal,digital,sources}/*.zig`. Each file declares
sized arrays such as `pub const input_complexfloat32 = [64]std.math.Complex(f32){ ... };`
and `pub const output_taps_128_cutoff_0_2_float32 = [64]f32{ ... };` (the serializer
emits fixed lengths like `[64]f32{...}` and `[128]std.math.Complex(f32){...}`, not
`[_]`).

They are produced by `generate.py` (105 lines, invoked via `zig build generate`),
which uses Python + NumPy + SciPy to compute the "ground truth" output for a given
input + block parameters, then rewrites the Zig source that the unit tests import.
Each vector file contains a recipe block delimited by `// @python` tags;
`generate.py` strips the leading `// ` from those lines, `exec`s the resulting Python
(which calls a `vector(name, value)` helper), truncates everything below the
`// @autogenerated` marker, and re-serializes the vectors. `complex64` arrays become
`std.math.Complex(f32)` literals, `bool` arrays become `[N]u1`, and `bytes` become
`[N]u8`. The random seed is fixed (`random.seed(1)`) for determinism and values are
rounded to `PRECISION = 8` decimals.

This is how the project maintains bit-for-bit (or within float epsilon) equivalence between:
- Pure Zig implementations
- liquid-dsp accelerated paths
- Any future VOLK/FFTW paths
- The mathematical definition of the algorithm

Every signal block test does something like (from `src/blocks/signal/lowpassfilter.zig`):

```zig
const BlockTester = @import("../../radio.zig").testing.BlockTester;
const vectors = @import("../../vectors/blocks/signal/lowpassfilter.zig");
...
var tester = try BlockTester(&[1]type{std.math.Complex(f32)}, &[1]type{std.math.Complex(f32)}).init(&block.block, 1e-6);
try tester.check(2, .{&vectors.input_complexfloat32}, .{&vectors.output_taps_128_cutoff_0_2_complexfloat32}, .{});
```

`BlockTester(input_data_types, output_data_types)` takes comptime *slices of types*
(one per port), and is initialized with `init(&block.block, epsilon)`. Its `check`
method signature is `check(rate, input_vectors, output_vectors, hooks)`, where the
vector tuples hold one const slice per port and `hooks` is `.{}` by default. Source
blocks use `checkSource`, and `process` returns the output tuple directly.

The tester drives the block through a `TestSampleMux` (from `core/sample_mux.zig`)
and compares each produced output port against the expected vector using
`expectEqualVectors` (in `core/testing.zig`), which applies the per-element float
epsilon passed to `init`.

This is industrial-strength DSP engineering hygiene.

## 3. Benchmark Infrastructure

`zig build benchmark` builds `benchmarks/benchmark.zig` (always `.ReleaseFast`) into a
`benchmark` exe. Each entry in its `BenchmarkSuite` table builds a small flowgraph,
calls `flowgraph.start()`, sleeps for `BENCHMARK_TRIAL_DURATION_MS` (= 5000, i.e. 5
seconds) via `std.Thread.sleep`, then `flowgraph.stop()`. Throughput is reported by a
`BenchmarkSink` (`src/blocks/sinks/benchmark.zig`), which every `report_period_ms`
(default 3000 ms) prints a line of the form
`[<title>] <n> <prefix>S/s (<n> <prefix>B/s)` — sample rate and byte rate with an SI
prefix (so e.g. `MS/s`/`MB/s` at megabyte scale), not a fixed `MS/s`/`MB/s` unit.

An optional command-line argument filters the suite by case-insensitive substring of
the benchmark name (`args[1]`).

This lets the maintainers quantify the impact of acceleration libs, compiler flags,
CPU affinity, etc., on end-to-end pipeline throughput.

The benchmarks deliberately use `ZeroSource` or `SignalSource` feeding expensive
blocks to stress data movement and compute. The heaviest case,
`benchmark_fir_filter_five_back_to_back`, chains five 256-tap `FIRFilterBlock`s
(real taps, complex input) back-to-back; other cases cover 128-tap FIRs, IIR filters,
FM de-emphasis, downsamplers, the frequency translator/discriminator, complex
magnitude, and the complex-to-real/imag converters.

## 4. Philosophy Embodied

- "Works everywhere, faster when possible."
- "Correctness is defined by reference DSP libraries (SciPy), not by us."
- "Measurement is part of the engineering loop (benchmarks + vectors)."

The optional nature of the accel libs + the generated vector tests together make the performance claims credible and the numerical quality high, without compromising the "single `zig build` " simplicity for new users.
